# Skeleton Replay PyCharm Plugin Handoff

## Current State

This repository is the private JetBrains/PyCharm frontend for Skeleton. It was
created as a separate repo from the Python engine so the plugin can use normal
Gradle/Kotlin/JetBrains release tooling while `../skeleton` continues to grow as
the PyPI library.

Repository:

- GitHub: `ml-affairs/skeleton-replay-plugin`
- Visibility: private
- Default branch: `main`
- Latest verified CI gate: `gradle test buildPlugin`
- Heavy Marketplace verification: documented but not run in normal CI because
  `verifyPlugin` exhausted GitHub hosted-runner disk during the first scaffold.

The current scaffold includes:

- IntelliJ Platform Gradle Plugin 2.x setup in `build.gradle.kts`
- PyCharm Professional target with bundled `PythonCore`
- `Skeleton` Tool Window registration
- Report, Workflow, Artifacts, Quality, and Log placeholder tabs
- Project settings for interpreter command, install command, output directory,
  include/exclude filters, max events, and report-opening preference
- `Replay with Skeleton` action for Python files
- Kotlin model classes for `session.json`
- Marketplace signing/publishing configuration through environment variables
- CI workflow for `gradle test buildPlugin`

Important limitation: the current action only prepares and displays the command.
It does not yet execute Skeleton, read `session.json`, or populate the workbench.

## Sibling Python Engine

The sibling repo is `../skeleton`.

`../skeleton` remains the source of truth for tracing, artifact generation, the
HTML report, and all Python package behavior. The plugin must not reimplement
tracing or parse raw events as its first integration surface.

The relevant engine release is:

- PyPI package: `skeleton-replay`
- Released version: `0.7.0`
- Release tag: `v0.7.0`
- GitHub release: `https://github.com/ml-affairs/skeleton/releases/tag/v0.7.0`

Version `0.7.0` introduced the IDE integration contract:

- each CLI/API run writes `session.json`
- `session.json` records:
  - `schema_version`
  - `skeleton_version`
  - `command`
  - reproducible `invocation`
  - `project_root`
  - traced `target`
  - artifact paths for `trace`, `snapshot`, `workflow`, `quality`,
    `quality_markdown`, optional `report`, and `session`
  - metrics for events, nodes, and edges
  - `target_exit_code`
  - `target_error`
  - `report_opened`

Plugin code should treat `session.json` as the stable discovery contract, then
load the linked artifacts. Do not scrape CLI console output for artifact paths.

## Intended Product Direction

The plugin should feel like an embedded architecture workbench inside PyCharm,
not just an artifact viewer.

The core user experience should center on the HTML report:

- render `report.html` inside a PyCharm Tool Window
- keep Workflow, Quality, Artifacts, and Log tabs as supporting surfaces
- let a user click a module, instance, function, or method inside the report
- map that report selection back to project source
- open the matching file in the editor
- navigate to the matching class/function/method where PSI resolution allows it
- select or reveal the file in the Project tree on the left

The report remains owned by the Python package. The plugin owns IDE integration:
launching Skeleton with the project interpreter, loading artifacts, embedding
the report, and bridging report selections to PyCharm navigation.

## Immediate Next Steps

1. Add a real Skeleton runner service.
   - Build a project service that constructs `python -m skeleton_replay run ...`
     and later `python -m skeleton_replay pytest ...`.
   - Use the configured interpreter command from `SkeletonSettings`.
   - Pass explicit `--project-root`, `--out-dir`, and `--no-open`.
   - Capture stdout/stderr into the Log tab.
   - Detect missing `skeleton_replay` and show an install action using the
     configured install command.

2. Load `session.json` after a run.
   - Read the configured output directory.
   - Parse `session.json` into the existing Kotlin data model.
   - Show artifact paths and target status in the Artifacts tab.
   - Load `workflow.md` and `architecture_quality.md` into their tabs.
   - Show target errors without hiding successfully generated partial artifacts.

3. Embed `report.html`.
   - Use JCEF when available.
   - Fall back to an external browser/open-file action when JCEF is unavailable.
   - Load the local `report.html` path from `session.json`, not from guessed
     output-directory conventions.

4. Add report-to-IDE selection bridge.
   - Extend the HTML report contract in `../skeleton` only if needed.
   - Preferred shape: report emits a selection event containing node id, event
     id, qualified name, module, file path if available, and line if available.
   - Plugin receives the event through JCEF JavaScript bridge.
   - Plugin resolves the source file and navigates using PyCharm PSI where
     possible.
   - If no exact PSI element is found, open the file and show the artifact
     metadata in the Tool Window.

5. Add tests around command construction and manifest loading.
   - Unit test runner command construction.
   - Unit test `session.json` parsing.
   - Keep full IntelliJ UI/integration testing for a later slice once the runner
     and report bridge exist.

## Suggested First Implementation Slice

Start with command execution plus manifest loading before deep JCEF work:

- create `SkeletonRunnerService`
- create `SkeletonSessionLoader`
- update `ReplayCurrentFileAction` to run the command instead of only displaying
  it
- update `SkeletonWorkbenchPanel` with methods to display run logs, artifacts,
  workflow text, quality markdown, and report path
- keep report rendering as a simple placeholder/open action for this slice

Then follow with the JCEF report embed and selection bridge as the second slice.
This keeps each PR reviewable and avoids mixing process execution, artifact
parsing, browser embedding, and IDE navigation all at once.

## Commands And Verification

Current normal plugin gate:

```bash
gradle test buildPlugin
```

Marketplace/release gate:

```bash
gradle verifyPlugin
```

`verifyPlugin` should be run locally or on a larger runner before Marketplace
publishing. It was removed from normal push CI because the GitHub hosted runner
ran out of disk during JetBrains verification.

Engine smoke command to use from a Python project with `skeleton-replay>=0.7.0`:

```bash
python -m skeleton_replay run --project-root <project-root> --out-dir <project-root>/.skeleton/pycharm/latest --no-open <python-file>
```

After it runs, the plugin should read:

```text
<project-root>/.skeleton/pycharm/latest/session.json
```

  whether the plugin can bridge existing report DOM state safely enough for v0.
